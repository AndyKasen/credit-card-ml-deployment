# Организация A/B-тестирования

## Концепция A/B-теста

Сравниваем текущую модель (v1, RandomForest) с новой версией модели (v2, возможно даже применим в ней другой алгоритм, например LogisticRegression) на реальных данных, затем принимаем решение о замене модели в production.

- **Контрольная группа (A):** запросы обрабатываются моделью v1
- **Тестовая группа (B):** запросы обрабатываются моделью v2
- **Распределение трафика:** случайное, 50% на группу A, 50% на группу B
- **Продолжительность:** 2 недели (минимально — 1 неделя для накопления статистики)

## Способ разделения трафика

При каждом запросе к эндпоинту **/predict** сервер случайным образом (с вероятностью 50%) выбирает, какую модель использовать. Выбор фиксируется в логах (поле **model_version**).

## Продолжительность 2 недели (минимум)

- Нужно собрать несколько тысяч запросов для правильных выводов
- Как раз 2 недели (а лучше месяц) можно будет отследить активность клиентов в будни и выходные.

## Метрики и анализ

### Основная метрика: F1-score для класса "дефолт" (1)

F1-score — среднее гармоническое между Precision и Recall. 

Формула: F1 = 2 × (Precision × Recall) / (Precision + Recall)

### Дополнительная метрика: Recall для класса "дефолт"

Recall (полнота) — доля реальных дефолтов, которые модель правильно предсказала.

Формула: Recall = TP / (TP + FN)

**Обоснование выбора Recall, а не Precision:**

Для банка пропустить дефолт (False Negative) значительно дороже, чем ошибочно отклонить хорошего клиента (False Positive). Recall показывает, сколько реальных дефолтов модель смогла обнаружить. Цель — максимизировать Recall, не допуская катастрофического падения Precision.

## Статистический анализ

### Применим Z-test для пропорций

Для сравнения метрик двух групп используется Z-test для независимых выборок. Этот тест проверяет гипотезу:

- **H_0 (нулевая гипотеза):** разницы между моделями нет, F1(v1) = F1(v2)
- **H_1 (альтернатива):** разница есть, F1(v2) > F1(v1)

Уровень значимости: alfa = 0.05.

### Доверительные интервалы

Для каждой метрики строится 95% доверительный интервал. Если интервалы двух моделей не пересекаются — разница статистически значима.

### Критерии успешности теста

Модель v2 признаётся лучшей, если выполняются **все три условия**:

1. F1-score (v2) > F1-score (v1) — основная метрика улучшилась
2. p-value < 0.05 — разница статистически значима
3. Recall (v2) ≥ Recall (v1) — новая модель не стала хуже находить дефолты

Если хотя бы одно условие не выполнено — в production остаётся модель v1.
