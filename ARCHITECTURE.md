## 1. Архитектура сервиса
### Монолит vs микросервисы

**Обосновать выбор архитектуры**
- Прежде всего, весь проект содержится в одном контейнере - это **монолит**. То есть весь сервис - приём запросов, ответ, предсказание, логирование работает внутри одного Docker-контейнера.
- Так как нам надо решать одну задачу - прогнозировать дефолт, то дробить на микросервисы проект нет смысла
- Можно одному человеку поддерживать весь код
- Запукается быстро, одной командой docker run
- Микросервисы понадобились бы, если бы множество сервисов банка обращались бы к модели и часто менялась бы логика обработки запросов.

### Концепт брокеров сообщений

Брокер сообщений — это программа-посредник, которая принимает сообщения от одних сервисов и доставляет их другим. Он работает асинхронно: отправитель не ждёт ответа, а кладёт задачу в очередь и идёт дальше.

**Пример RabbitMQ**

| Компонент | Назначение |
|-----------|------------|
| **Producer** | Отправляет сообщения (например, Flask при запросе на предсказание) |
| **Exchange** | Принимает сообщения и раскладывает по очередям по правилам маршрутизации |
| **Queue** | Очередь сообщений, хранит задачи до обработки |
| **Consumer** | Забирает сообщения из очереди и обрабатывает (например, делает предсказание) |

**Как брокер очередей на примере RabbitMQ мог бы использоваться в нашем проекте**

| Сценарий | Описание |
|----------|----------|
| **Асинхронные предсказания** | Клиент отправляет запрос, Flask кладёт его в очередь 'predict_queue' и сразу отвечает "принято". Consumer достаёт задачу,обрабатывает моделью и сохраняет результат в базу |
| **Логирование** | Каждый запрос создаёт сообщение в очереди 'log-queue'. Отдельный Consumer собирает логи и пишет в файл или базу данных, не замедляя основной сервис |
| **Устойчивость** | Если сервис Consumer упал — сообщение остаётся в очереди, а другой Consumer подхватит его|

### Логирование и мониторинг

### 1.

Сервис записывает логи в двух местах:
- **Файл `logs/api.log`** — JSON-записи, по одной строке на событие
- **Терминал** — удобно смотреть при запуске из терминала и в случает отладки

### 2

Каждое событие — одна строка в формате JSON.

**Типы событий:**

| Событие | Описание | Пример |
|---------|----------|--------|
| `startup` | Сервис запущен, модель загружена | `{"event": "startup", "message": "Модель и scaler загружены", "timestamp": "2026-06-05T10:00:00"}` |
| `request` | Поступил запрос к `/predict` | `{"event": "request", "endpoint": "/predict", "features_count": 23, "timestamp": "2026-06-05T10:01:00"}` |
| `prediction` | Модель вернула результат | `{"event": "prediction", "prediction": 1, "probability": 0.67, "timestamp": "2026-06-05T10:01:01"}` |
| `error` | Ошибка валидации запроса | `{"event": "error", "error": "Expected 23 features, got 5", "timestamp": "2026-06-05T10:02:00"}` |

### Мониторинг в production (ELK-стек)

В production-системе, когда серверов много и у каждого свой файл логов, то целесообразно собирать их централизованно.

**ELK-стек** (Elasticsearch + Logstash + Kibana) — стандартное решение для сбора и анализа логов:

| Компонент | Функция |
|-----------|---------|
| **Logstash** | Собирает логи со всех серверов (читает файлы или принимает по сети) |
| **Elasticsearch** | Хранит логи и обеспечивает быстрый поиск |
| **Kibana** | Веб-интерфейс с графиками и поиском по логам |

**Функции системы мониторинга ELK:**

| Метрика | Что можно мониторить в нашем задании |
|---------|---------------|
| Количество запросов в минуту | Оценка нагрузки на сервис |
| Распределение предсказаний (0/1) | Контроль дрейфа модели |
| Средняя вероятность дефолта | Обнаружение изменений в данных клиентов |
| Количество ошибок валидации | Выявление проблем на стороне клиентов API |
| Время ответа | Контроль производительности |
